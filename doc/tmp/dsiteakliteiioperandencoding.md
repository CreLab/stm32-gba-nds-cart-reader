# DSi TeakLite II Operand Encoding


**Syntax Notes**
The official Teak syntax specifies all operands in \"source,dest\
order, that\'s opposite of most other ASM languages which us
\"dest,source\" order (except 68000 processors, which the Teak i
apparently inspired on). One exception are instructions tagged
\"NoReverse\" in the opcode list: These ones do have the operand
ordered as how one would usually expect them.
Operands tagged as \"Implied xx\" are effectively used by the hardware
although the official syntax doesn\'t specify them in source code. O
the other hand, operands tagged as \"Bogus xx\" are specified i
official source code syntax, although the hardware doesn\'t actually us
that operands in that form.
The nocash (dis-)assembler syntax reverses the operand order (except
those flagged as NoReverse), removes Bogus operands, and inserts Implied
operands. Moreover, immediates and memory operands are specified
differently\...

**Memory Operands**


**Operand Encoding**
Below shows the binary encoding for registers/conditions. For example,
\"Ab@10\" in the opcode list can be \"b0,b1,a0,a1\" encoded in bit10-11
Some instructions re-use the same bitfield for multiple operands (eg.
when performing two operations on the SAME operand, or when expecting
DIFFERENT operands: for \"Ax@12,Ax@not12\" one of the registers mus
\"a0\", and the other must be \"a1\").

**offs and step**
- Memory operands with \"offs\" allow to read from \[Rn\], \[Rn+1\], o
- \[Rn-1\]. Operands with \"step\" allow to increment/decrement register
(for old TL opcodes this is usually specified alongsides with memory
operands, for new TL2 opcodes it\'s typically specified as separat
\"\|\|\" instruction (eg. \"\|\| Rn@0stepZIDS@3\"; which can be omitte
if the step is zero).
The official syntax wants \"+1,-1\" abbreviated to \"+,-\" in some case
(but not abbreviated in other cases). step \"+s\" does probably refer t
\"stepi or stepj\", but it\'s rather unclear which one, maybe stepi i
used for r0..r3, and stepj for r4..r7, or vice versa\... or maybe i
depends on each opcode (particulary opcodes that allow to use \"Rn\
(r0..r7) might use the same step in ALL cases).
